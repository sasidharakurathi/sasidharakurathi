### 🧠 Problem Solving Profile

*How I think, debug, and build — drawn from real project experience:*

<details open>
  <summary><b>🐛 Debugging Style</b></summary>
  <br/>
  <p>
    My debugging approach is systematic and evidence-driven, combining real-time log analysis with async-aware instrumentation. In <b>RoboSwift</b>, I traced runtime failures by parsing raw <code>stdout</code> streams from spawned child processes via Tauri's command API, surfacing errors immediately in the UI rather than waiting for a crash. In <b>VirtuHire-AI-Development</b>, I debugged concurrent async workers by instrumenting each pipeline stage independently — isolating bottlenecks between the transcription, scoring, and report-generation queues. For <b>AI-Interviewer</b>, I reproduced WebSocket signaling failures by replaying session state through Django Channels logs, pinpointing exactly where connection handshakes broke under load. When confronting <b>LLM hallucinations in ATS</b>, I built a client-side retry-and-validation wrapper that catches malformed or incomplete JSON outputs, re-prompts the model with tightened schema constraints, and only persists results once a structurally valid response is confirmed.
  </p>
</details>

<details open>
  <summary><b>🚀 Approach to New Projects</b></summary>
  <br/>
  <p>
    I start every project by defining a tight scope, choosing the lightest viable architecture, and automating the setup so that anyone can clone-and-run immediately. For <b>venvdrop</b>, this meant a zero-dependency, pure-Python design — no third-party packages, just the standard library — so the tool itself never becomes a dependency problem. With <b>Project-T</b>, I sketched the cross-platform architecture first (Tauri + Rust core, universal TCP tunneling) before writing a single line of UI code, ensuring the low-level networking contract was solid. For <b>Axon-File-Manager</b>, I designed the multi-stage AI pipeline (BM25 → Vector Search → RRF Fusion → Re-ranking) on paper before implementation, treating each stage as an independently testable module. In <b>Route Memory</b>, I adopted a feature-first Flutter architecture with Riverpod from day one, preventing state-management sprawl as new GPS and cloud-sync features were added. Every project ships with comprehensive setup scripts (PowerShell/Bash) so onboarding takes minutes, not hours.
  </p>
</details>

<details open>
  <summary><b>📚 How I Learn</b></summary>
  <br/>
  <p>
    I learn by building production-grade systems around unfamiliar technology, not by following tutorials in isolation. To master <b>RAG</b>, I implemented it end-to-end in <b>Repo-Scribe</b> — local documentation indexing, smart diff batching, and Gemini-powered generation — without a vector database, forcing a deep understanding of context retrieval. I taught myself <b>computer vision</b> by training a custom <b>YOLOv8</b> model from scratch in <b>glove_burger_detection</b>, writing the video-to-frame extraction, dataset splitting, and real-time inference scripts myself. For <b>AI-Interviewer</b>, I tackled <b>WebRTC</b> signaling and <b>DeepFace</b> emotion/presence detection simultaneously, iterating on live video sessions to understand the interplay between latency and model accuracy. Deploying <b>Ollama</b> locally for both <b>ATS</b> and <b>AI-Interviewer</b> gave me hands-on experience with the trade-offs of local vs. cloud LLM inference — cold-start times, VRAM constraints, and quantization quality.
  </p>
</details>

<details open>
  <summary><b>⚖️ Decision Making</b></summary>
  <br/>
  <p>
    Every technology choice I make weighs the lightweight/heavyweight trade-off against long-term maintainability and security requirements. I chose <b>Tauri + Rust</b> for <b>RoboSwift</b> specifically because Electron would have added ~150 MB of runtime overhead; Tauri produces a ~10 MB binary with native system access and memory safety by default. I default to <b>FastAPI</b> for backend services because its async-first design, Pydantic validation, and automatic OpenAPI docs cut boilerplate while keeping performance close to raw ASGI throughput. For <b>prompt-polish</b>, I adopted a <b>BYOK (Bring Your Own Key)</b> model stored in <code>chrome.storage.local</code>, deliberately avoiding a backend server — users own their credentials, and there is no central point of data exposure. In security-sensitive contexts (ATS biometric login, AI-Interviewer proctoring), I reach for <b>multiprocessing isolation</b> and modification-based caching over simpler thread-based approaches, accepting higher complexity for stronger process-boundary guarantees.
  </p>
</details>
